# F-006 — Deterministic hooks · Specification

## Goal

Turn ASPIS's operational rules into **deterministic walls and light automation**
that fire without an LLM. A first cut (the old "guards") forced one asset kind to
edit six core files; this rebuild rides the F-005 asset-kind + capability core, so
every piece is a drop-in.

Two surfaces, each with one job:

1. **Git hooks** — fire at the *commit boundary*. The enforcement wall (block bad
   commits) plus light auto-run (refresh context, maintain `.gitignore`, clean junk).
2. **Runtime scope-guard** — fires at the *tool-use boundary* inside the agent loop,
   per runtime (Claude / OpenCode). Blocks an out-of-scope `Edit`/`Write` *before* it
   ever reaches a commit.

Both are deterministic, fast, and auditable. Neither runs the test suite.

## Non-goals

- No tool-use **tracing** / JSONL event spine (that is Phase 4).
- No LLM call anywhere in a hook.
- No rewrite of the lifecycle-hook system (`.aspis/hooks/<event>/`, `hooks.py`). Git
  hooks install into git's own `.git/hooks/`; all hook *logic* lives under the existing
  main `.aspis/scripts/` folder — no new top-level `.aspis` directory is invented.
- The committer agent and the full git workflow are **F-007** (next feature); F-006
  ships only the mechanism the committer will lean on.

## User stories

- **US-1 — The wall holds.** As the system, when an agent (or a human) tries to commit
  files outside the active feature's scope, touch an R-009 protected path, leak a
  secret, or write a non-conformant commit message, the commit is **blocked** with a
  clear reason — no reliance on the model behaving.
- **US-2 — The repo stays clean on its own.** As a maintainer, when I commit, junk
  (shell-redirect ghosts, stray garbage) is removed automatically, new file kinds are
  reflected in `.gitignore`, and the context brain (registry / code-map / state /
  changes) is refreshed — with no manual step. If nothing is junk and nothing is new,
  the scripts do nothing.
- **US-3 — Caught early.** As an agent mid-task, when I try to edit a file outside my
  feature's allowed scope, the runtime stops the edit immediately with the reason,
  instead of letting me discover it at commit time.
- **US-4 — One command to arm it.** As a user, a single install step (run by `init`)
  writes the hook wrappers into `.git/hooks/`, pointing them at the version-controlled
  logic in `.aspis/scripts/hooks/`. Idempotent — safe to re-run.

## Functional requirements

### Git hooks (commit boundary)
- **FR-001**: An idempotent installer writes thin wrappers into `.git/hooks/`
  (`pre-commit`, `commit-msg`, `post-commit`) that `exec` the matching Python entry
  point under `.aspis/scripts/hooks/`. The logic is version-controlled in `.aspis`;
  the wrappers in `.git/hooks` are (re)generated by the installer on `init`.
- **FR-002**: `pre-commit` **blocks** on: (a) a staged file outside the active feature's
  `scope.allowed` or inside `scope.forbidden`; (b) an R-009 protected path without an
  approval marker; (c) a detected secret; (d) a fast lint/format failure
  (`ruff format --check` + `ruff check` on staged Python). It **never** runs pytest.
- **FR-003**: `commit-msg` **blocks** any message that violates `commit-convention.yaml`
  — wrong/absent type, scope not `F-NNN[/T-NN | /T-NN..T-MM]`, subject over the max
  length, or any forbidden-attribution token (`co-authored-by`, `claude`, …). The
  convention data is the single source; the hook adds no inline rules.
- **FR-004**: `post-commit` **never blocks**; it runs the light auto-run scripts and
  exits 0 even on their failure (warn only).

### Auto-run scripts (light, fast, idempotent)
- **FR-005**: **One** housekeeping handler that (a) removes junk files (shell-redirect
  ghost names, configured garbage) when present, and (b) deletes a folder's `.gitkeep`
  the moment that folder also holds a real file. It **does nothing** when the tree is
  already clean. Callable standalone and from `pre-commit`.
- **FR-006**: A `.gitignore` maintainer detects the project stack (reusing
  `detect.detect_stack`, stack overridable) and writes the **canonical** ignore body
  from the Toptal gitignore API (`…/developers/gitignore/api/<stack>`, which requires a
  valid stack keyword). It works **offline** from a bundled template cache when the
  network is absent, updating the cache when online. Exposed as a CLI command
  (`aspis gitignore [stack]`) and reused by the auto-run step. It manages only its own
  marked block in `.gitignore`; a no-op once current.
- **FR-007**: A context-refresh step runs the existing `.aspis/scripts/context/*`
  (registry, code-map, state, changes) — reused, not reimplemented.

### Runtime scope-guard (tool-use boundary)
- **FR-008**: A single shared `scope_guard` script decides whether a path is in the
  active feature's scope; it is reused by both the git `pre-commit` and the runtime
  hooks (one source of truth for "is this in scope?").
- **FR-009**: Claude is wired via `.claude/settings.json` (`PreToolUse` on
  `Edit|Write`, blocking via exit code 2). OpenCode is wired via
  `.opencode/plugins/scope-guard.ts` (`tool.execute.before`, blocking via `throw`).
  Both invoke the same shared script.
- **FR-010**: Runtime-hook wiring is **per-runtime** and emitted by each runtime
  adapter (the adapter owns where its files land — D-002), gated by the capability
  model: a runtime that does not declare the capability is simply skipped.

### Packaging / consistency
- **FR-011**: Git-hook entry points and scripts ship as ordinary **brain** asset kinds
  (zero `assetkinds.py` change — F-005 Story 1); the base profile lists them; the
  generated runtime is regenerated so ASPIS dogfoods its own hooks.
- **FR-012**: The full existing test suite stays green; new tests cover the blocking
  decisions (scope, message, secret), the no-op behaviour of the auto-run scripts, and
  capability-gated runtime-hook emission.

## Success criteria
- **SC-1**: A staged out-of-scope file fails `pre-commit`; an in-scope clean change passes.
- **SC-2**: A `feat(F-006/T-02): …` message passes; `wip`, a bare `update`, or a message
  containing `Co-Authored-By` fails `commit-msg`.
- **SC-3**: Running the junk handler on a clean tree changes nothing and exits 0; on a
  tree with a ghost file it removes exactly that file.
- **SC-4**: After install, `.git/hooks/pre-commit` exists and a commit fires the hooks;
  re-running the installer is a clean no-op.
- **SC-5**: Export to a Claude-only profile emits `.claude/settings.json` and no
  `.opencode` plugin; the OpenCode plugin appears only for OpenCode.

## Feature rules and style
- Lean, data-driven, small files; logic in Python, rules in `config/*.yaml`.
- DRY: one scope decision, one junk handler, reuse `detect_stack` and the context scripts.
- New behaviour = new files; no edit to F-005 core except adding adapter capability data.
