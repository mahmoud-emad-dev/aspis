## 1. The 3-layer architecture

ASPS is a **file-first, OpenCode-centered, multi-model software production factory** that
**dogfoods itself** — Project #1 is the system itself, and the system builds the system.

### 1.1 The four-layer transform (the big picture)

Defined in `ARCHITECTURE.md §2–3` and `docs/catalog.md`. Every artefact is a deterministic
transform between four layers:

```
┌──────────────────────────────────────────────────────────────────┐
│ 1. FACTORY  (this repo)                                          │
│    catalog of agents/, skills/, commands/, profiles/, templates/, │
│    hooks/, scripts/   +  rules/  +  ARCHITECTURE.md  +  plans    │
│    → the single source of truth                                  │
└──────────────┬───────────────────────────────────────────────────┘
               │  transform + render (per runtime)
               │  asps init-project / regenerate / promote
               ▼
┌──────────────────────────────────────────────────────────────────┐
│ 2. PROJECT BRAIN  .asps/                                         │
│    features/, traces/, reviews/, lessons/, context/,             │
│    current/active_feature.json, current/active_run.json,         │
│    runtime.lock.*.json                                           │
│    → durable, tool-neutral, runtime-portable state               │
└──────────────┬───────────────────────────────────────────────────┘
               │  generated by the engine
               ▼
┌──────────────────────────────────────────────────────────────────┐
│ 3. RUNTIMES  .opencode/  .claude/                                │
│    agents/, skills/, commands/, plugins/, tools/, hooks/,        │
│    settings/config                                              │
│    → generated, replaceable, byte-parity-guarded                 │
│    → NEVER hand-edit (the totality guard enforces this)          │
└──────────────┬───────────────────────────────────────────────────┘
               │  runs
               ▼
┌──────────────────────────────────────────────────────────────────┐
│ 4. PRODUCT  src/  tests/  docs/  configs/  …                     │
│    the project being built                                       │
└──────────────────────────────────────────────────────────────────┘
```

### 1.2 The 3-layer agent roster (D-026, refined by D-028)

The **agent roster** is the human-facing delegation hierarchy inside the factory and
the exported runtimes. It is **3 layers + 2 shared helpers + 1 bootstrap agent**:

```
L1  project-lead  ........... primary (the ONLY agent the human talks to)
                                 │
L2  asps-system-lead  ........ primary · owns agents/skills/templates/configs/commands/hooks
     asps-planning-lead  ..... primary · drives the planning kit (P0–P9)
     asps-build-lead  ....... primary · owns implementation, runs the gate first
     asps-reviewer  ......... primary · read-only verdict, never reviews own work
     ── 4 L2 primaries (D-021: every lead is `mode: primary`) ──
     asps-fix-lead  ......... subagent lead · gate/test failure → minimal fix
     asps-tester  ........... subagent lead · testing strategy, red→green
     lead-researcher  ....... subagent lead · keeps reference knowledge current
     ── 3 L2 subagent leads (D-028: still 2-level run depth) ──
                                 │
L3  workers (leaves — "Delegates to: none"):
     planning:  clarify, idea-capture, task-decomposer
     build:     python-builder, api-builder
     review:    security-reviewer, commit-reviewer
     fix:       bug-triager, gate-fixer
     test:      test-author
     system:    asps-opencode-author, asps-claude-author, committer, governance
     research:  codebase-explorer, docs-fetcher, web-researcher
     ── (database-builder, prd-writer, architecture-reviewer, gate-runner,
         coverage-analyst, refactor-builder, frontend-builder, docs-builder,
         config-builder, style-reviewer, prd-writer, scope-guard, project-report-
         explorer, error-recorder, dependency-scout, plan-architect, spec-writer,
         acceptance-designer, template-author, skill-author, config-author,
         command-author, regression-guard, trace-discipline  →  22 ARCHIVED
         per D-026; folded into leads via skills, or deferred to P5/P7) ──

shared helpers (any agent may invoke):
     project-explorer  ........ read-only repo / pattern search
     context-feeder  .......... returns task-scoped context (Level 1–4)
     context-summarizer  ...... condenses traces → .asps/context

bootstrap (transient, export-only, self-deletes):
     asps-bootstrap  .......... fresh export → live project in one chat turn
                                 (D-B1: extracted from asps-system-lead, runs
                                 once, removes itself when done)
```

**D-021, D-026, D-028 together settle the run depth:** every Level-1 lead can be
talked to directly (`mode: primary`), and may delegate to Level-2 subagents. The
original "third delegation level" (a subagent calling another subagent) is
**parked** — it failed on OpenCode and the owner prefers a 2-level model anyway
(D-028).

### 1.3 The factory vs product split (D-B4)

A foundational invariant: the **factory is heavy** (the big reusable catalog +
the `asps` Python engine), the **product is lean** (a profile-selected subset
of the catalog + the generated runtime + the actual code). Profiles gate the
export so a tiny project never inherits the whole 88-asset catalog.

The **asps-bootstrap** agent is **export-only** — it ships into user projects and
self-deletes. The factory is already bootstrapped; the package is inert there
(export-scope), never re-runs.

## 2. The 9 laws (R-001…R-009) in practice

Source: `rules/SYSTEM_RULES.md` (the frozen text, mirrored by `ARCHITECTURE.md §5`,
enforced by `tests/test_law_config_drift.py`). They are the only rules; every
agent, skill, template, standard, review, hook, and task packet references them
by ID.

| ID | Rule | What it means in practice |
|---|---|---|
| **R-001** Scope | Only modify files the task or active feature allows. Never touch forbidden files. | The `feature.yaml` lists `scope.allowed` and `scope.forbidden`; `commit_guard.py` and `gate-fixer` enforce it; tasks with no allowed-files list are rejected. |
| **R-002** Gates first | Run deterministic gates (lint/format/types/tests + scope/secret checks) before declaring work done. | Five universal gates (ruff-format, ruff-check, mypy, pytest, byte-parity totality); also `scope-check`, `secret-check`, `validate-runtime`, `validate-index`. Gates run locally, in pre-commit, and in CI. A reviewer is never asked before gates pass. |
| **R-003** Stable prompts | No casual mid-run rewrites of `AGENTS.md`, core skills, or system rules. Durable changes need an improvement proposal + human approval. | The catalog is versioned; agents are not auto-tuned; rule changes go through the `improvement` proposal flow. |
| **R-004** One writer | One active writer per branch/worktree. Reviewers are read-only. | Only `committer` may run `git commit`; the reviewer is explicitly `edit: deny` in its permission intent; parallel work uses separate branches/worktrees. |
| **R-005** Tests-as-spec | Tests are part of the spec. Behaviour changes need tests. Bug fixes need regression tests. Never delete or weaken a test to pass a gate. | The anti-test-gaming rules block: deleting tests, weakening assertions, skipping failing tests without approval, changing acceptance mid-task, marking done without gates. |
| **R-006** Teach | When a stronger model catches a weaker model's miss, capture the lesson and propose a durable improvement. | The `lesson → improvement proposal → human approval → update rule/skill/template/test/agent/routing → commit` loop (`dashboard/LESSONS.md`, L-001…L-007). |
| **R-007** Trace | Every important step, tool action, gate result, review, decision, and failure emits a trace event. No silent runs. | `asps trace-event`, `trace-hook`, `trace-commit`, the `trace-pipe` shim, the OpenCode plugin, and the `.asps/traces/{raw,runs}/` JSONL files. |
| **R-008** Pin models | Once model IDs are verified, every subagent uses an explicit model. Do not let subagents silently inherit an expensive primary model. | Each agent's `## Abstraction` block has a `Model intent:` line; the exporter writes the pinned `model:` into the runtime YAML. |
| **R-009** Human gate | Architecture, rules, standards, security-sensitive changes, model-routing changes, and self-improvement promotions require human approval. | R-003 + R-009 together define the **constitutional surface** — only `governance` may edit `rules/**` and only after explicit human approval (enforced by code, not prose). |

A 10th implicit rule appears in `rules/SYSTEM_RULES.md §Local helper-prompt workflow`:
the `local/`, `_local/`, `*.local.md`, `CLAUDE.local.md` paths are git-ignored and
**must never be committed** — they exist for private helper prompts, scratch
research, and disposable context.

- The clean **context-ladder** discipline (`AGENTS.md` → `.aspis/context/CURRENT_STATE.md`
  → `.aspis/context/RECENT_CHANGES.md` → `.aspis/index/CODE_MAP.md` →
  `FILE_REGISTRY.yaml`).
- The **`.opencode/skills/` structure** (the opencode-native skill format) already

- A much **smaller, more honest starting point**: the current ASPIS does not
  carry the 22-archived-agents baggage, the god-file `cli/main.py`, or the
  Python↔shell dual-stack duplication (Critical Review A1, A4).
- **Mode-scaled planning** in spirit (the mode hints in F-016's SPEC/PLAN/TASKS
  templates: vibe / mvp / production) — OLD ASPS had F-027-PLANNING-UPGRADE as
  a *draft*, never executed.

### 4.4 The single biggest gap

**The `asps` engine and its CLI verbs are missing.** The current ASPIS is a
catalog + a context ladder + some skills, but the deterministic spine —
`validate-runtime`, `validate-index`, `doctor`, `proof`, the gate runner, the
totality guard, the golden manifest, the exporter, `promote` with its lock
classification (ADD / UPDATE / UNCHANGED / PROTECT / CONFLICT / UNTRACKED) — is
all roadmap, not code. The current ASPIS is "as if a 50-agent ASPS got
rebooted on day 0 with a clean context ladder and no engine." Re-introducing
even a thin `asps` CLI (status, validate-*, doctor, proof) is the highest-leverage
move available.

## 5. The closest thing to REFACTOR-ANALYSIS (the critical review)

The brief asked for `local/guide/REFACTOR-ANALYSIS.md`. The artefact that plays
this role is **`docs/CRITICAL_REVIEW_2026-06-10.md`** — a 255-line, evidence-based
self-audit run on 2026-06-10. Here is the synthesised finding list (the "11
findings" of the brief map onto the review's 10 prioritised backlog items + the
11 architectural findings A1–A4, R1–R4, P1–P4, S1–S4, T1, U1–U3, Gap 1–3):

### 5.1 Architecture (the highest leverage)

1. **A1 — Dual-stack duplication (Python engine vs shell scripts). 🔴** The same
   capabilities exist twice (detect, validate, junk, gates, status, trace) in
   `src/asps/*.py` (15.4 kLOC) and `scripts/*.ps1|.sh` (~23 files). Two code paths,
   two bug surfaces, guaranteed drift. Fix: declare the boundary
   ("shell = portable project runtime; Python = factory engine") and have each
   call the other.
2. **A2 — Exported slash-commands call a CLI that isn't in the export. 🔴** In a
   real external project, `python -m asps start-feature` fails (ModuleNotFoundError).
   The practical face of A1 — the #1 "works in the factory, breaks in the field" gap.
   Fix: re-implement the lifecycle verbs as shell, or ship `asps` as a pip dep.
3. **A3 — Two init engines with a contract that the code violates. 🟠** `initializer.py`
   and `bootstrap.ps1` overlap — both git-init, both write `AGENTS.md` (different

## 7. The operating model (12 design principles)

The brief asked for `ASPS_OPERATING_MODEL.md` (the 12 design principles). The file
**does not exist** in the working tree, and the 12 principles are not enumerated
in any single file. They are reconstructed below from `ARCHITECTURE.md §1, §2, §3,
§13, §14, §15, §17, §22, §26`, from `docs/architecture.md`, and from the
6-section discipline in `local/phases/00-README.md`. The 12 principles that
emerge from the captured source:

1. **File-first, no hidden state.** Every artefact is a file. The catalog plus
   the locks fully determine the runtimes; the gates prove it on every change.
2. **Spec-driven, task-packet-driven, test-gated, reviewed, traceable,
   self-improving, multi-model, runtime-portable.** (`ARCHITECTURE.md §1` lists
   these 8 properties; the 12th below is a synthesis of how they interact.)
3. **Four layers, not one.** Factory → project brain → runtimes → product.
   Each layer is **derived** from the previous, never hand-edited.
4. **Quality = model capability × task clarity × test strength × review
   discipline.** The prime directive. ASPS invests in the *non-model* factors
   so cheaper models can safely do production work.
5. **Deterministic first.** Any check that *can* be a script *must* be a script.
   LLM judgment is reserved for genuinely open-ended work — planning and review
   *content* — where there is no deterministic oracle.
6. **Single source of truth for every rule and engine.** One policy engine, one
   trace writer, one bootstrap path, one catalog validator, one law/config
   source. A test proves the copies (Python + shell, or archive + live) never
   disagree.
7. **Narrow role per agent.** Two agents must not overlap. A narrow role is
   easier to trust, route to, pin a model for (R-008), and improve.
8. **Specialize first, extract shared later.** Do not create `common-*` skills
   too early. Give each lead the specific skills it needs; extract shared
   pieces only after real repetition shows.
9. **Anti-test-gaming.** Block: deleting tests to pass, weakening assertions,
   skipping failing tests without approval, changing acceptance mid-task,
   marking done without gates. Tests are part of the spec (R-005).
10. **Catalog is truth, runtime is derived, never hand-edited.** Edit the
    catalog; run `asps regenerate`; `asps promote --write`; the totality guard
    proves the live runtime reproduces byte-for-byte.
11. **Gates before review, review before trace, trace before improvement.**
    R-002 → reviewer → R-007 → R-006. The order is the loop:
    `idea → spec → plan → tasks → build → tests/gates → review → trace → lesson → improvement`.
12. **Self-improvement is gated by humans.** R-003 + R-009: no auto-rewrites of
    `AGENTS.md`, `rules/SYSTEM_RULES.md`, `ARCHITECTURE.md`, security standards,
    or model routing for high-cost models without explicit human approval. No
    self-improvement automatically mutates the constitution.

The 6-part phase discipline in `local/phases/00-README.md` is the human-facing
manifestation of these 12: every phase file answers *what / why / tasks /
options rejected / testing / effect on the real project* in the same order.
The **"build user-first; build the floor first, then the door, then the rooms"**
line in `00-README.md` is the rule for sequencing the work itself.

## 9. The bootstrap contract

The "bootstrap" is the first chat in a fresh project. The plan lives in
`local/phases/BOOTSTRAP-UX-PLAN.md` (Phase 2 of the Refactor Plan). The
contract is:

### 9.1 The flow (the target experience)

1. **Detect** — empty vs populated, `.git`, stack/framework, runtimes present,
   OS/IDE. One structured detection record (reuses `detect.py` / `detect_env.py`).
2. **Confirm or skip** — show detected name/stack/runtime; user may override,
   but **never block** (`--yes` non-interactive). Bounded autonomy (D-1): enrich
   from detected facts only, never invent.
3. **Run** `asps bootstrap --write` — brain seed, hooks, **enrich** (fill
   AGENTS.md name/definition/stack slots; expand `.gitignore` from stack),
   manifest, first commit, doctor.
4. **Promote 5 primaries** — `project-lead`, `asps-planning-lead`,
   `asps-build-lead`, `asps-reviewer`, `asps-system-lead`. Leave
   `fix-lead`/`tester`/`lead-researcher` as subagents.
5. **Self-cleanup (post-bootstrap)** — remove bootstrap-only assets from the
   **product** runtime (`asps-bootstrap` agent, `asps-bootstrap` skill/workflow/
   templates, shell orchestrator scripts), and write `bootstrap.done` (timestamp
   + engine hash) into `manifest.json`. Hand control to `project-lead`.
6. **Thereafter** — no agent references bootstrap; `--check` short-circuits on
   `bootstrap.done`.

### 9.2 The 6 decisions (D-B1…D-B6)

- **D-B1** Dedicated `asps-bootstrap` agent, not system-lead. Single
  responsibility; lets the whole package self-delete cleanly.
- **D-B2** Self-deleting package. Bootstrap is one-time; leaving its
  agent/skill/workflow/scripts invites agents to re-check forever and bloats
  the product. Cleanup removes them and marks `bootstrap.done`.
- **D-B3** Exactly 5 primaries. Small primary surface = simpler UX, less drift.
  Fix/test/research stay reachable as subagents under build/review. **R-009
  approved.**
- **D-B4** Factory ≠ Product. The bootstrap package is **export-only**: it
  ships into user projects and self-deletes there. The factory is already
  bootstrapped and keeps the package inert/source-only (export-scope), never
  re-running it.
- **D-B5** No guards in standing agents. Only the transient `asps-bootstrap`
  agent knows about bootstrap. `project-lead` / `asps-system-lead` carry zero
  bootstrap prose. Once gone, nothing re-asks.
- **D-B6** Bounded autonomy / namespacing. Enrich only from detected facts +
  templates; write only under `.asps/`/`.opencode/`/`.claude/`/`AGENTS.md`/
  `.gitignore`; never the user namespace, never the parity-protected generated
  bodies.

### 9.3 The 7 build units (B-1…B-7)

B-1 author `asps-bootstrap` agent (catalog) · B-2 strip bootstrap from
`asps-system-lead` · B-3 five-primary promotion (R-009) · B-4 enrichment fills

## 11. Lessons learned (L-001…L-007)

Source: `dashboard/LESSONS.md` (7 named lessons, newest first, all
2026-06-10). Each captures what broke, why, and the durable improvement
proposal. The lessons are the most distilled "what we learned" the OLD system
captured.

### L-001 — Real `git` in logic tests is non-deterministic under load
- **What happened.** `test_doctor.py`'s silent-run tests spawned real `git`
  in temp repos. In a full-suite run on Windows the cumulative subprocess
  load made `git` intermittently fail to spawn — 3–6 tests flaked while all
  passed in isolation. First blamed orphaned `opencode.exe`; killing them
  did not fix it. Root cause was the suite's own git load.
- **Lesson.** A test that verifies *logic* must not depend on a real external
  binary that can fail for environmental reasons.
- **How to apply.** Inject the external result (stub `_run_git`); keep one
  focused integration test for the real wrapper. Harden the *production* path
  with retries, but never make green/red hostage to spawn luck. (→ D-023)

### L-002 — Catalog↔live drift: never hand-edit the derived runtime
- **What happened.** The P2 lane couldn't `promote` cleanly, so it hand-copied
  skills into `.opencode/` → 60 skill dirs (30 real + 30 duplicate category
  dirs), a runtime lock describing a non-existent state, and permission edits
  living only in live files. The exporter flattens skills, so the category
  dirs were never something `promote` produces.
- **Lesson.** `.opencode/` is derived and disposable; the catalog is truth.
- **How to apply.** Integrate a lane by merging its catalog + `src` + `scripts`
  and re-deriving the runtime; regenerate the lock as a faithful snapshot;
  propagate any live-only edits back into the catalog *before* the next
  `promote` or they get erased. Lanes never `promote`; the integrator does,
  once. (→ D-022)

### L-003 — "Done" in a lane is not green-and-clean
- **What happened.** Both lanes reported all tasks "done" in `TASK_STATUS.md`.
  Reality: P4's HEAD was red (stale asset-count assertions), its green-making
  fix sat uncommitted, and it had littered junk files (`-`, `Copy`, `=6.100`).
  P2 had baked duplicate skills and a wrong lock.
- **Lesson.** Self-reported status is a claim, not a fact.
- **How to apply.** On integration, always verify the tree (`git status`), run
  the gates yourself, and check acceptance criteria — don't trust the status
  file. (→ D-024)

### L-004 — Asset-count assertions are a fragile but useful drift canary
- **What happened.** Adding one command (`proof`) bumped the runtime asset
  count 86→87→88 and broke hardcoded `assert report.checked == N` tests in
  three places; the P2 merge conflicted on the same number (it claimed 125
  because of the duplicate skills).
- **Lesson.** Hardcoded counts are brittle, but they *did* catch the
  duplicate-skill bloat.
- **How to apply.** Keep the count assertion (it's a cheap canary) but always
  derive the expected number from the real composition (agents + flat skills
  + commands) and update it deliberately, never to accommodate bloat. (The
  golden manifest `tests/golden/catalog_assets.txt` + `refresh-golden.py
  --check` is the durable replacement.)

### L-005 — One Python env + one `.git` across worktrees is a shared-state trap
- **What happened.** A lane ran `pip install -e .`, repointing the global
  `asps` editable install to its worktree — so `import asps` resolved to the
  wrong source everywhere, and a lane's `pytest` could silently test the
  wrong tree. Direct evidence appeared in a session export
  (`Set-Content ..._editable_impl_asps.pth`). Separately, all worktrees
  share one `.git`.
- **Lesson.** Parallel worktrees share the Python env and the git object
  store.
- **How to apply.** Override per-run with `PYTHONPATH=<repo>/src`; give each
  worktree its own venv (the durable fix); back up refs before branch ops;
  one branch per worktree. (memory: shared-editable-install-hazard,
  parallel-terminal-git-hazard)

### L-006 — The richest trace signal is the OpenCode session *export*, not the DB
- **What happened.** The old `ingest_opencode_session` chased per-session
  JSONL that no longer exists; the live DB (`opencode.db`) only has aggregates
  + part counts. The `session export` JSON (UTF-16, `{info, messages[]}`) is
  a full transcript: per-message agent + model + time, reasoning, every tool
  call's args+results, and real patch diffs.
- **Lesson.** Use the export for deep analysis; the DB for cheap aggregates.
- **How to apply.** The `analyze-session` track ingests exports → attribution
  + tool-error + delegation + junk-command signals, linked to our events
  and git diff. Two real sessions showed build agents working *directly*
  (zero `task` delegations) — concrete evidence of the L3-delegation gap.
  (memory: opencode-session-export)

### L-007 — Garbage files come from agent bash output-redirects
- **What happened.** Junk files (`-`, `=6.100`, `=2.5`, `Copy`, `edit`)
  appeared at repo root — the P4 lane (which built the junk guard) littered
  them. Session analysis pinned the class: bash commands with `>` /
  `Set-Content` / `New-Item` whose target got mangled.
- **Lesson.** The junk guard catches them at commit, but the root is the
  agent's shell redirect.
- **How to apply.** The deterministic fix is auto-clean (strengthen `junk.py`
  patterns + run it automatically); true prevention needs the per-agent
  bash-permission pass (deferred, D-025). The `analyze-session` junk-command
  detector identifies the exact offending command for later blocking.

### The cross-cutting lessons (synthesis)

1. **Drift is a silent bug.** Every rule, engine, and copy must live in
   *exactly one place* with a machine-checked test that fails if the copies
   diverge. (L-002, L-004, L-005.)
2. **Self-reported status is a claim, not a fact.** Integrate by verifying the
   tree, the gates, and the acceptance — not the status file. (L-003.)
3. **Hermetic tests beat realistic tests for logic; reserve real-binary
   tests for the production wrapper.** (L-001.)
4. **The richest signal lives in the session export, not the DB.** (L-006.)
5. **Junk files are a symptom of shell-redirect accidents; prevention is a
   permission problem, detection is a junk-guard problem.** (L-007.)
6. **Shared infrastructure across parallel worktrees is a trap.** (L-005.)
7. **The 2-level delegation model is the safe default; the 3-level model is
   parked until the OpenCode L3 failure is solved.** (D-028, L-006's evidence.)

## 12. The complete vision — what the COMPLETE version of ASPS would look like

Synthesised from `ARCHITECTURE.md §26` (final principle), `local/phases/00-README.md`

### 12.1 The final principle (verbatim from `ARCHITECTURE.md §26`)

> "ASPS should make software production increasingly deterministic.

> ```text

> human reviews more

### 12.3 The complete ASPS, end-to-end

A person installs ASPS once. The `asps` command works everywhere. The engine
**knows its surroundings** (OS, runtimes, tools, stack). A single
`asps bootstrap` takes an empty folder to a *live, self-explaining* ASPS
project in one chat turn — detect, confirm, run, promote 5 primaries, self-clean,
hand to `project-lead`.

The user talks to `project-lead`. `project-lead` classifies the request,
holds the human gate, and delegates to one of the L2 leads. Each lead runs
a deterministic spine: planning produces a build-ready plan (P0–P9, scaled
to the working mode), build produces a gated commit, review produces a
read-only verdict, fix produces a minimal patch, test produces evidence.
Every step emits a trace event.

The catalog is truth; the runtimes are derived; the totality guard proves
it on every change. The five universal gates (ruff-format, ruff-check,
mypy, pytest, byte-parity) run before any "done" claim. The trace spine
captures the raw firehose (every tool call), the curated events (start,
close, commit, gate result, hand-off, lesson), and the session ingestion
(OpenCode export + Claude JSONL → attribution, tool errors, delegation
patterns, junk commands, cost).

The lesson loop catches a stronger model correcting a weaker model's miss,
captures the lesson, and proposes a durable improvement — a new rule, a new
skill, a new template, a new test pattern, a new routing choice. R-003 +
R-009 ensure no self-improvement auto-mutates the constitution. The human
approves the high-level decisions; the system enforces the rest.

The dashboard is a read-model over a stable trace store: the active feature,
the next action, the recent runs, the gate health, the recent commits, the
lessons, the per-agent cost. The web UI is a thin layer over the same data.

The harvest path lets new patterns land safely: record the source, check the
license, adapt to ASPS format, prove in a lab, review, promote, add to a
profile, trace. External MCP servers, plugins, and harvested skills are
treated as untrusted until reviewed.

The complete ASPS, in one line, is: **a file-first, dogfooded, multi-model
software production factory whose prime directive is `Quality = model
capability × task clarity × test strength × review discipline`, whose
constitution is the 9 laws (R-001…R-009), whose planning is the 10-phase
kit (P0–P9), whose execution is the 3-layer agent roster (1 L1 + 7 L2 + 20
L3 + 3 helpers + 1 transient), whose truth is the catalog and whose
runtimes are derived, whose evidence is the five universal gates + the
totality guard + the trace spine, whose growth is the lesson loop, and
whose hand is the human gate.**

### 12.4 What the complete ASPS would feel like to use

- **Day 0 — install.** `uv tool install --from <repo> asps`; `asps doctor`
  reports 0 FAIL. The engine is pinned to a known version.
- **Day 1 — start a project.** `asps init-project --target ./myapp --profile
  python-cli --write`; `asps bootstrap --target ./myapp --write`; the
  project is live, the brain is seeded, the runtimes are generated, the
  git hooks are installed, the 5 primaries are promoted, the bootstrap
  package has self-deleted. `asps doctor` → 0 FAIL.
- **Day 2 — first feature.** `asps start-feature F-001-hello-world`;
  talk to `project-lead`; it classifies → routes to `asps-planning-lead`;
  planning produces a SPEC/PLAN/TASKS scaled to "vibe" (1 line intake, no
  spec, no clarify, no research, 1-para plan, no task packets, no
  plan-critic, self-approve). `asps-build-lead` implements against the
  gate; `asps-reviewer` returns approve-with-notes; `asps-tester` writes
  a regression test; the trace records every step.
- **Day 3 — second feature.** `asps start-feature F-002-auth`; this time
  "mvp" mode kicks in: short spec, ≤3 clarify questions, light research,
  yes-plan, yes-tasks, quick plan-critic, human approve at P9. The
  packets are scoped enough that a cheap model implements unaided.
- **Week 2 — production feature.** `asps start-feature F-010-billing`;
  "production" mode: full SPEC, full ≤5-Q clarify, full research + dated
  proofs, full architecture (R-009 human gate), full feature map, scoped
  packets, full plan-critic, human gate at P9. The plan-critic catches
  three over-engineering points before build starts.
- **Quarter 1 — the loop is boring.** Cheap models do 80% of the
  implementation. The reviewer is the second pair of eyes. The trace
  shows per-feature cost. The lessons loop has proposed 4 durable
  improvements; 2 were approved and shipped; 1 was rejected; 1 is in
  review. The harvest path added 2 skills from external sources (both
  reviewed, both proved in a lab, both adapted to ASPS format). No
  critical-review findings remain open. `asps doctor` is always 0 FAIL.
  The dashboard is the cockpit.

### 12.5 What would still be hard

Even the complete ASPS leaves some things hard — these are the
**honest limitations** captured in the lessons + the critical review:

- **Cheap-model work still needs small scoped tasks with clear allowed
  /forbidden files.** The system cannot make a vague request succeed; it
  can only make a precise one succeed cheaply.
- **Human judgment stays load-bearing for architecture, rules, security,
  model routing, and self-improvement.** R-009 is not a tax; it is the
  design.
- **3-level delegation on OpenCode is parked (D-028).** A subagent that
  delegates to another subagent fails on OpenCode. The 2-level model is
  the safe default until the L3 failure is solved.
- **The session-export trace is the richest signal, not the DB.** Tool
  calls (L-006) miss the OpenCode `event` bus (Gap 1) — `file.edited`,
  `session.idle/created/error`, `todo.updated`, `permission.updated/
  replied`, `command.executed`, `vcs.branch.updated`, `chat.message`. The
  plugin needs to be expanded post-merge.
- **The dual-stack duplication (Python engine vs shell scripts) is the
  single biggest structural risk (Critical Review A1).** Even with the
  boundary declared, two stacks will drift unless the seam is enforced.
- **The 12 design principles are real but are not yet a single document.**
  They are *embedded* in `ARCHITECTURE.md` and the 6-part phase
  discipline. Extracting them into a canonical `ASPS_OPERATING_MODEL.md`
  is a deferred, valuable piece of work.

## 13. Quick-reference: the top 10 things to carry forward to ASPIS

1. **The 3-layer agent roster (1 L1 + 7 L2 + 20 L3 + 3 helpers + 1 transient)**
   is the right shape. ASPIS's `.opencode/agents/` and `.opencode/skills/`
   already follow the same idea; align the catalog to the `agents/<domain>/
   <name>.md` + `## Abstraction` block contract.
2. **The 9 laws (R-001…R-009) survive intact.** They are the constitution;
   no rewrite needed.
3. **The four-layer transform (factory → brain → runtime → product) with
   "catalog is truth, runtime is derived"** is the load-bearing invariant.
   Introduce a thin `asps` CLI (status, validate-*, doctor, proof) and the
   totality guard as the first move.
4. **The narrow-role principle + the `## Abstraction` block** is the right
   agent contract. Body says *which* skill; skill says *how*.
5. **The five universal gates** (ruff-format, ruff-check, mypy, pytest,
   byte-parity) are the bar. Bring them in as the first concrete artifact.
6. **The golden manifest** (`tests/golden/catalog_assets.txt` + a refresh
   script + a `--check` test) replaces fragile count snapshots and is a
   10-line import.
7. **The trace spine** (raw + curated + session ingest) is the high-value
   long-term investment. Start with the curated `trace-event` + the
   `trace-commit` (post-commit hook) and a JSONL appender; add the OpenCode
   plugin and the session ingest in a later pass.
8. **The bootstrap contract** (transient `asps-bootstrap` agent, 5-primary
   promotion, self-cleanup, `bootstrap.done` marker, bounded autonomy to
   `.asps`/`.opencode`/`.claude`/`AGENTS.md`/`.gitignore` only) is the
   cleanest first-day experience. Build it before the catalog grows.
9. **The 7 lessons (L-001…L-007) are the distilled "what we learned."**
   Bake them in as the default behaviour: hermetic tests for logic, single
   source of truth for every rule + engine, integrate by verifying not by
   trusting, golden manifest for drift, per-worktree venv, session export
   for deep analysis, auto-clean for junk.
10. **The prime directive is the design.** `Quality = model capability ×
    task clarity × test strength × review discipline.` Every product
    decision ASPIS makes should pass through this equation. The non-model
    factors are the leverage.
