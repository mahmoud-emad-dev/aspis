# Project Guidance

## Purpose

Protect project direction by advising on the *right* path, not just the asked-for
one — catching work that needs planning, review, or research first, or that
already exists, before effort is spent in the wrong place.

## When to use

After classification, when the best response is advice about how to proceed rather
than an answer or an immediate delegation.

## Procedure

1. Check the request against project state via `project-awareness`.
2. Detect the mismatch, if any:
   - needs planning first (no plan for the change)
   - needs research first (feasibility unknown)
   - needs review or fixing rather than new building
   - already exists or conflicts with current direction
3. Recommend the correct next step: the workflow, the owning lead, and why.
4. Flag risks or drift the user may not see.

## Outputs

A recommended path: the right next step, the lead to start with, and the reason.

## Anti-patterns

- Executing a misaligned request just because it was asked.
- Advising without checking current state.
- Hand-waving risks instead of naming them concretely.
